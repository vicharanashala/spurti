# Contributing to Spurti

Thank you for contributing! This guide covers everything you need to get started.

## Quick Start

```bash
# 1. Clone the repo
git clone https://github.com/amanraj74/spurti-iit-ropar-vled-.git
cd spurti-iit-ropar-vled-

# 2. Install dependencies
npm install
npm --prefix client install

# 3. Copy and configure environment
cp .env.example .env
# Edit .env вҖ” defaults work for local dev with ALLOW_STUDENT_SEARCH=true

# 4. Start MongoDB locally (required)
# Docker:
docker run -d -p 27017:27017 --name spurti-mongo mongo:8

# 5. Build client
npm run build

# 6. Start server
npm run dev
# Server runs at http://localhost:5290

# 7. Run tests
npm test
```

## Key Concepts

### Authentication
Students authenticate via the `chatengine_token` cookie (set by Samagama). The server forwards this to Samagama's internal auth endpoint to get the student's email.

Admin endpoints are protected by `X-Admin-Email` + `X-Admin-Token` headers.

### SP Transactions
`SPTransaction` is an append-only ledger. `Student.totalSp` is pre-computed and kept in sync. Never recalculate SP from raw data in the web app вҖ” the pipeline does this.

### Session Labels
The pipeline generates session labels like `Day 1 (16 May)` or `Orientation (15 May)`. The `server/config.js` session labels (`'15 May Morning'`, etc.) are for the old ingestion system and are deprecated. Do not use them for new features.

## Common Tasks

### Adding a new API endpoint

1. Add the route in `server/server.js`
2. Use `validateBody()` or `validateQuery()` from `server/utils/validators.js` for input validation
3. Use `adminGuard` middleware if it's an admin endpoint
4. Import shared utilities from `server/utils/`

### Adding a new utility function

If your function could be used in multiple places, add it to `server/utils/`:
- `utils/email.js` вҖ” email normalization, masking
- `utils/parse.js` вҖ” CSV, date, and Zoom date parsing
- `utils/validators.js` вҖ” Zod validation schemas

### Writing tests

Tests live in `server/__tests__/`. Run them with `npm test`.
- Unit tests for utilities: `server/__tests__/utils/`
- Unit tests for services: `server/__tests__/services/`

Example:
```javascript
import { myFunction } from '../utils/myModule.js';

describe('myFunction', () => {
  test('does something', () => {
    expect(myFunction('input')).toBe('expected');
  });
});
```

## Code Style

- ES Modules (`import`/`export`, `"type": "module"` in `package.json`)
- No semicolons ( ASI style)
- 2-space indentation
- Prefer `const` over `let`, never use `var`
- Use `.then()`/`.catch()` or `async`/`await` consistently

## Before Submitting a PR

1. **Run tests:** `npm test` вҖ” all must pass
2. **Build client:** `npm run build` вҖ” must succeed
3. **Lint check:** `npm run lint` вҖ” no errors
4. **Write tests** for any new utility or service functions
5. **Keep commits focused** вҖ” one logical change per commit
